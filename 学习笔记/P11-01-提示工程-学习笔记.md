## 一句话总结

Prompt 是你给 AI 写的"任务说明书"。10 种模式是写说明书的 10 种策略。你先选策略，再填内容，最后跑测试验证效果。

---

## 核心认知

**Prompt 不是"聊天技巧"，是给 200 亿参数模型写的精确指令。**

- AI 不是人，不会猜你没说的话
- 说得越清楚，结果越接近你想要的
- 同一个模型，模糊 Prompt 和精确 Prompt 的输出质量天差地别

### 类比理解

| Prompt 概念 | 生活类比 |
|------------|---------|
| Prompt 本身 | 去餐厅点菜——"来点吃的" vs "宫保鸡丁，微辣，不要花生，配米饭" |
| System Message | 告诉服务员"你是米其林主厨"——设定标准和身份 |
| Few-Shot | 给服务员看两道菜的照片——"按这个风格做" |
| Chain-of-Thought | 让厨师"先想食材和顺序，再动手"——分步推理 |
| Temperature | 创意旋钮——0=完全确定，0.7=有点变化，1.0=随机创意 |

## Prompt 的三段式结构

```
System Message → User Message → Assistant Prefill
（设定身份规则）   （具体任务）     （引导输出格式）
```

| 组件 | 作用 | 类比 |
|------|------|------|
| System Message | 设定模型身份、规则、约束，对所有请求生效 | Spring 的 `@Configuration` |
| User Message | 具体的任务或问题 | `@RequestBody` |
| Assistant Prefill | 提供模型响应的前几个 token 来引导格式 | `@ResponseBody` 控制输出格式 |

**关键**：大多数人只用 User Message，忽略了 System 和 Prefill。这就像写 API 只写 Controller，不写全局配置和序列化器。

## 10 种 Prompt 模式

模式不是模板，是**设计策略**。你拿到任务，先想"需要什么策略组合"，再写 Prompt。

### 决策流程

```
拿到任务
  → 要不要设角色？         → Persona
  → 要不要给示例？         → Few-Shot
  → 要不要指定输出结构？   → Template Fill
  → 要不要分步推理？       → Chain-of-Thought
  → 要不要设行为边界？     → Guardrail / Boundary
  → 要不要让它自己检查？   → Critique
```

### 10 种模式详解

| # | 模式 | 一句话 | 适用场景 | 温度 |
|---|------|--------|---------|------|
| 1 | **Persona** | 告诉它"你是谁" | 所有任务（基线策略） | 0.7 |
| 2 | **Few-Shot** | 给 2-3 个输入输出示例 | 格式要求高的任务（提取、分类） | 0.0 |
| 3 | **Chain-of-Thought** | 要求先想再做 | 数学、逻辑、多步骤推理 | 0.3 |
| 4 | **Template Fill** | 按模板提取信息 | 数据提取、结构化输出 | 0.0 |
| 5 | **Critique** | 自己检查再改进 | 高质量要求的输出（报告、公开内容） | 0.5 |
| 6 | **Guardrail** | 定规矩划范围 | 用户面向的产品 | 0.3 |
| 7 | **Meta-Prompt** | 让 AI 帮你写 Prompt | 新任务的 Prompt 开发 | 0.7 |
| 8 | **Decomposition** | 拆解复杂问题 | 3-7 个子问题的复杂任务 | 0.3 |
| 9 | **Audience Adaptation** | 按受众调整表达 | 同一内容面向不同人群 | 0.5 |
| 10 | **Boundary** | 硬拒绝越权请求 | 合规场景（金融、医疗） | 0.0 |

### 常见组合

| 任务 | 模式组合 | 预期准确率 |
|------|---------|-----------|
| 数据提取 | Template Fill + Few-Shot | 95%+ |
| 用户问答 | Persona + Guardrail + Boundary | 90%+ |
| 代码生成 | Persona + Chain-of-Thought | 85%+ |
| 内容生成 | Persona + Critique（双 pass） | 90%+ |
| 分类 | Few-Shot + Guardrail | 95%+ |

## Temperature（温度）

控制输出的随机性。

| 温度 | 效果 | 适用场景 |
|------|------|---------|
| 0.0 | 确定性，每次一样 | 数据提取、分类、代码 |
| 0.3 | 保守，基本可预测 | 摘要、分析 |
| 0.7 | 平衡 | 通用问答 |
| 1.0 | 创意，每次不同 | 头脑风暴、创意写作 |
| 1.5+ | 混乱 | 永远别在生产用 |

**规则**：用 Temperature 或 Top-p，不要同时用两个。

---

## 4 个反模式（踩坑警告）

| 反模式 | 问题 | 类比 |
|--------|------|------|
| Prompt Injection | 用户输入覆盖你的 System Prompt | SQL 注入 |
| 过度约束 | 规则太多，模型没余力做任务 | 接口文档 50 页，调用方直接放弃 |
| 矛盾指令 | "简洁"和"详尽覆盖每个边界"冲突 | `@Size(min=100, max=50)` |
| 假设模型特定行为 | ChatGPT 上好用 ≠ Claude 上好用 | Tomcat 能跑 ≠ Jetty 能跑 |

## 实战示例：从任务到 Prompt

**任务**：从用户评价里提取情感和关键信息，输出 JSON。

**第 1 步：选模式** → Template Fill（主）+ Few-Shot（辅助）+ temp=0.0

**第 2 步：组装**

```
System: 你是数据提取引擎。只输出 JSON，不要任何解释。

User:
从以下用户评价中提取信息，填入模板。

模板：
{
  "product": "产品名称",
  "sentiment": "positive/negative/mixed",
  "pros": ["优点列表"],
  "cons": ["缺点列表"],
  "rating": "1-10"
}

示例 1：
输入："小米 14 拍照很好，但电池不太耐用"
输出：{"product": "小米 14", "sentiment": "mixed", "pros": ["拍照好"], "cons": ["电池不耐用"], "rating": "6"}

示例 2：
输入："ThinkPad X1 键盘手感一流，性能也强，就是太贵了"
输出：{"product": "ThinkPad X1", "sentiment": "positive", "pros": ["键盘手感好", "性能强"], "cons": ["价格高"], "rating": "8"}

现在提取这条评价：
输入："AirPods Pro 降噪效果惊艳，佩戴舒适，但连接偶尔断"
```

**第 3 步：测试** → 跑 5 条数据，检查 JSON 合法性、字段完整性、有无废话

## 关键术语

| 术语 | 通俗说法 | 精确定义 |
|------|---------|---------|
| System message | "指令" | 高优先级消息，设定身份、规则、约束 |
| Temperature | "创意旋钮" | 对 logit 分布的缩放因子 |
| Top-p | "核采样" | 限制 token 采样到累积概率超过 p 的最小集合 |
| Few-shot | "给例子" | Prompt 中包含 2-10 个输入输出示例 |
| Chain-of-thought | "分步思考" | 让模型展示中间推理步骤 |
| Prompt injection | "越狱" | 用户输入覆盖 System Prompt 的攻击 |
| Context window | "能读多少" | 模型单次能处理的最大 token 数 |
| Assistant prefill | "开头引导" | 提供响应前几个 token 来引导格式 |

---

## 跨模型兼容性

| 模式 | 跨模型一致性 | 说明 |
|------|------------|------|
| Few-Shot | 非常高 | 示例在所有模型上都有效 |
| Template Fill | 非常高 | 明确结构，模型差异小 |
| Chain-of-Thought | 高 | 所有主流模型都支持"分步思考" |
| Persona | 高 | 不同模型对角色具体程度的响应不同 |
| Guardrail | 中等 | Claude 最严格，GPT 长对话可能偏离 |
| Boundary | 中低 | 拒绝行为因模型而异，需逐个测试 |
